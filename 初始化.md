# 业务逻辑驱动型 Harness — Superpowers Blueprint 完整实施计划

## 定位

在 `d:\codeai\blueprintharness` 下构建一套**独立于任何具体项目**的业务开发 Harness 蓝图。
它不是文档仓库，而是一台**行为约束引擎**——通过 Iron Laws、HARD-GATE、Red Flags 表、Rationalization Prevention 等 Superpowers 验证过的行为塑造手段，让 AI Agent 无法绕过业务逻辑直接写代码。

---

## 设计原则（提炼自 Superpowers 14 个 Skill）

| 原则 | 来源 Skill | 在本 Harness 中的体现 |
|------|-----------|---------------------|
| **Iron Laws** — 一句话不可违背的铁律 | verification, debugging | `AGENT.md` 顶部的绝对禁令 |
| **HARD-GATE** — 阶段门禁 | brainstorming | 每个工作流阶段必须满足前置条件才能进入下一阶段 |
| **Red Flags 表** — 反模式识别 | debugging, verification | Agent 自查表，发现自己正在做表中行为时必须 STOP |
| **Rationalization Prevention** — 借口粉碎表 | verification | "借口 → 现实" 对照表，预防 Agent 找理由跳过流程 |
| **Evidence Before Claims** — 证据先于断言 | verification | 任何"完成"声明必须附带运行证据 |
| **No Placeholders** — 禁止占位符 | writing-plans | 任务计划中不允许 TBD、TODO、"后续补充" |
| **Fresh Context** — 隔离上下文 | subagent-driven | 每个任务获得干净的、精心策划的上下文 |

## 各文件详细设计

### 1. [NEW] `AGENT.md` — 行为宪法

这是整个 Harness 的灵魂。参考 Superpowers 的 `verification-before-completion` 和 `systematic-debugging` 的写法风格，采用 **Iron Law + HARD-GATE + Red Flags + Rationalization Prevention** 四层防线。

#### 1.1 Iron Laws（铁律）

```
NO CODE WITHOUT BUSINESS ANCHOR.
NO COMPLETION WITHOUT EVIDENCE.
NO CONFLICT WITHOUT ESCALATION.
```

三条铁律分别对应：
- 代码必须包含 **精确的** `// @blueprint-ref` 锚点（需包含文档标题、规则编号或节点 ID）
- 完成声明必须附带 `validate.sh` 的执行输出
- 文档冲突必须上报人类，禁止自行判断

#### 1.2 文档权威等级（法定仲裁序列）

```
DOMAIN_MODEL.md  >  BUSINESS_PROCESS.md  >  PRODUCT_SENSE.md  >  ARCHITECTURE.md
（实体定义最高）    （流程定义次之）         （场景感知第三）      （技术骨架最低）
```

当 `DOMAIN_MODEL` 定义 Order 状态为 `PENDING`，而 `BUSINESS_PROCESS` 写成 `WAIT_PAY` 时：
- Agent 必须 **STOP**
- 以 `DOMAIN_MODEL` 为准提出冲突报告
- 等待人类搭档裁决

#### 1.3 工作流阶段门禁（HARD-GATE）

```
Phase 0: 启动 → 必须阅读 PRODUCT_SENSE.md + DOMAIN_MODEL.md + memory/lessons.md (经验库)
Phase 1: 需求理解 → 输出 specs/ 下的规格书，等待人类审批
Phase 2: 任务分解 → 输出 tasks/active/ 下的原子任务，包含“相关经验模式”扫描
Phase 3: 编码实现 → 每个代码块上方必须有 @blueprint-ref 精确锚点
Phase 4: 验证收尾 → 执行 validate.sh (包含正则深度检查)，通过后回答"防呆双问"
Phase 5: 归档沉淀 → 任务迁移至 completed/，经验写入 memory/
```

**HARD-GATE 规则：不得跳过任何 Phase。Phase N 未完成时，严禁执行 Phase N+1 的任何动作。**

#### 1.4 Red Flags 表（反模式自查）

| 如果你发现自己正在… | 立即 STOP，做什么 |
|-------------------|------------------|
| 没看 DOMAIN_MODEL 就开始写 Service 层代码 | 回到 Phase 0，阅读领域模型 |
| 说"我已阅读"但无法引用原文 | 重新打开文档，找到具体段落 |
| 同时修改 3 个以上文件 | 拆分任务，当前任务过大 |
| 觉得"这个改动太简单不需要规格书" | 最简单的规格书只需要 3 句话，但必须写 |
| 在代码中写了 `// TODO: 后续对齐业务流程` | 现在就对齐，不存在"后续" |
| 文档 A 和文档 B 说法不一致，你选了一个继续干 | 停下来，拉人类裁决 |
| 任务完成了但没跑 validate.sh 就想标记完成 | 跑完再说 |

> [!CAUTION]
> **红旗中断上报 (Iron Rule)**：一旦触发 Red Flags 表中的任何条目，必须**立即暂停**所有代码编辑动作，并在对话中以 `🚨 BLUEPRINT RED FLAG TRIGGERED` 开头，向人类搭档报告具体触发了哪一条以及补救计划。获得批准前禁止继续。

#### 1.5 Rationalization Prevention（借口粉碎表）

| 借口 | 现实 |
|------|------|
| "业务逻辑太简单，不需要引用" | 简单逻辑也有隐含的业务规则。引用是证据，不是负担 |
| "已经理解了，不需要看文档" | 理解 ≠ 引用。没有引用就没有可审计的证据链 |
| "文档还没写完，先写代码后补" | 先有蓝图后有建筑。没有文档的代码是技术债务 |
| "时间紧迫，跳过规格书" | 没有规格书的代码返工率 > 60%，实际更慢 |
| "这是技术重构，不涉及业务" | 任何改动都可能影响业务行为。至少检查波及分析 |
| "validate.sh 上次跑过了" | 上次 ≠ 这次。每次声明完成都需要当次的证据 |

#### 1.6 原子任务标准

单次 Task 的规模控制在整个特性开发预估工作量的 **1/5 ~ 1/10**。
- 足够小：Agent 不会进入上下文混乱的"发散状态"
- 足够大：包含完整的逻辑语义，不至于碎片化到无法审查

#### 1.7 变更影响分析（Impact Analysis）约束

修改任何 `harness/docs/` 下的核心文档时，必须在提交前输出一份"变更影响清单"：

```markdown
## 变更影响清单
- 修改文档: DOMAIN_MODEL.md — Order 实体新增 `PARTIAL_REFUND` 状态
- 可能受影响:
  - [ ] BUSINESS_PROCESS.md — 退款流程时序图是否需要新增分支？
  - [ ] VERIFY.md — 验收标准是否需要覆盖部分退款场景？
  - [ ] 相关 specs/ — 是否有进行中的规格书引用了旧状态？
```

人类搭档复核通过后方可继续。

#### 1.8 强制收尾——"防呆双问"

每个 Task 归档至 `completed/` 之前，Agent 必须在任务文件末尾回答：

```markdown
## 收尾反思
### Q1: 本次实现与业务文档有出入吗？
> [回答。如果有，摘录至 trace/failures.md]

### Q2: 本次是否发现了可复用的模式？
> [回答。如果有，摘录至 memory/lessons.md]
```

### 2. [NEW] `harness/docs/` — 业务蓝图库

#### 2.1 `DOMAIN_MODEL.md` 模板

包含以下标准章节：
- **实体关系图**（Mermaid erDiagram）
- **状态枚举与状态机**（Mermaid stateDiagram-v2）
- **全局业务规则**（编号规则列表，如 `BR-001: 订单总价不得为负`）
- **值对象定义**（如 Money、Address 等不可变对象）

#### 2.2 `BUSINESS_PROCESS.md` 模板

包含以下标准章节：
- **核心业务时序图**（Mermaid sequenceDiagram，必须包含成功路径 + 至少一个失败路径）
- **状态扭转矩阵**（表格：当前状态 × 事件 → 目标状态）
- **异常处理策略**（每个关键节点的失败后果和补偿动作）

#### 2.3 `PRODUCT_SENSE.md` 模板

- **业务价值画布**：目标用户、核心痛点、解决方案假设
- **用户旅程地图**：关键触点和情绪曲线
- **高风险业务点**：标记为 `⚠️ HIGH-RISK` 的业务环节

#### 2.4 `ARCHITECTURE.md` 模板

- **逻辑分层**：L0-L4 及依赖方向（允许/禁止）
- **核心组件图**（Mermaid graph）
- **技术选型记录**：选型理由和备选方案

#### 2.5 `VERIFY.md` 模板

- **按业务场景组织的验收清单**，每条使用 checkbox `- [ ]` 格式
- **冒烟测试路径**：最基本的端到端路径
- **边界条件测试**：极端值和异常输入

### 5. [NEW] `harness/scripts/validate.sh` — 机械化执法脚本

```bash
#!/bin/bash
# Blueprint Harness — 机械化验证脚本
# 用途：在任务声明完成前强制运行，拦截不合规的代码变更

PASS=true

# ═══════════════════════════════════════
# Probe 1: @blueprint-ref 探针
# 检查暂存区中的 .java/.kt/.py/.ts 文件是否包含业务锚点注释
# ═══════════════════════════════════════
echo "🔍 [Probe 1] 检查 @blueprint-ref 精确锚点..."
STAGED_FILES=$(git diff --cached --name-only --diff-filter=ACM | grep -E '\.(java|kt|py|ts|js)$')
if [ -n "$STAGED_FILES" ]; then
  for file in $STAGED_FILES; do
    # 深度检查：必须包含 DOMAIN_MODEL/BUSINESS_PROCESS 或 BR- 编号
    if ! grep -qE "@blueprint-ref:.*(DOMAIN_MODEL|BUSINESS_PROCESS|BR-[0-9]+)" "$file"; then
      echo "  ❌ FAIL: $file 缺少有效的 @blueprint-ref 引用（需指向具体文档或规则 ID）"
      PASS=false
    fi
  done
  if [ "$PASS" = true ]; then
    echo "  ✅ 所有变更文件均包含业务锚点引用"
  fi
else
  echo "  ⏭️  暂存区无代码文件，跳过"
fi

# ═══════════════════════════════════════
# Probe 2: 核心文档完整性
# ═══════════════════════════════════════
echo "🔍 [Probe 2] 检查核心文档完整性..."
REQUIRED_DOCS=("harness/docs/DOMAIN_MODEL.md" "harness/docs/BUSINESS_PROCESS.md" "harness/docs/PRODUCT_SENSE.md" "harness/docs/ARCHITECTURE.md" "harness/docs/VERIFY.md")
for doc in "${REQUIRED_DOCS[@]}"; do
  if [ ! -f "$doc" ]; then
    echo "  ❌ FAIL: 缺少核心文档 $doc"
    PASS=false
  else
    echo "  ✅ $doc"
  fi
done

# ═══════════════════════════════════════
# Probe 3: 任务收尾检查
# 确认 active/ 中的任务已包含收尾反思
# ═══════════════════════════════════════
echo "🔍 [Probe 3] 检查活跃任务收尾状态..."
for task in harness/plans/tasks/active/*.md; do
  [ -f "$task" ] || continue
  if ! grep -q "## 收尾反思" "$task"; then
    echo "  ⚠️  WARN: $task 缺少收尾反思章节"
  fi
done

# ═══════════════════════════════════════
# 最终裁决
# ═══════════════════════════════════════
echo ""
if [ "$PASS" = true ]; then
  echo "═══════════════════════════════════════"
  echo "  ✅ ALL PROBES PASSED"
  echo "═══════════════════════════════════════"
  exit 0
else
  echo "═══════════════════════════════════════"
  echo "  ❌ VALIDATION FAILED — 请修复后重试"
  echo "═══════════════════════════════════════"
  exit 1
fi
```

## 实施阶段

| Phase | 产出 | 验证方式 |
|-------|------|---------|
| **Phase 1** | 创建完整目录骨架（含所有空目录） | `tree` 命令确认结构 |
| **Phase 2** | 撰写 `AGENT.md`（含 Iron Laws、HARD-GATE、Red Flags、Rationalization Prevention 全部内容） | 人工走查：检查是否达到"LLM 无法忽视"的显著程度 |
| **Phase 3** | 填充所有文档模板（`DOMAIN_MODEL.md`、`BUSINESS_PROCESS.md`、`PRODUCT_SENSE.md`、`ARCHITECTURE.md`、`VERIFY.md`、`specs/_TEMPLATE.md`、`tasks/active/_TEMPLATE.md`） | 确认每个模板含引导注释和 Mermaid 示例 |
| **Phase 4** | 编写 `validate.sh`（含 3 个 Probe） | 模拟一段无 `@blueprint-ref` 的文件变更 → 脚本报错；加注释后 → 脚本通过 |
| **Phase 5** | 初始化 `trace/failures.md` 和 `memory/lessons.md`（含格式模板） | 检查文件存在且格式正确 |

## 验证计划

### 自动化验证
1. 运行 `bash harness/scripts/validate.sh`，期望在无代码变更时全部 PASS
2. 创建一个不含 `@blueprint-ref` 的测试文件并 `git add`，重新运行，期望 Probe 1 FAIL
3. 加上注释后重新运行，期望 ALL PROBES PASSED

### 人工走查
1. 检查 `AGENT.md` 的 Iron Laws 和 Red Flags 表是否足够醒目（使用大号标题、表格、代码块组合）
2. 确认所有模板文档的引导注释不为空白占位符
3. 检查目录结构和所有文件链接的有效性
